## MAJOR Findings (3)

### M1 — Falsy Zero on All ENV Config Values (Lines 151-155)

Five consecutive falsy zero instances. All five ENV config values use || instead of ??. If any ENV var is explicitly set to "0", Number("0") returns 0, || treats it as falsy, and the default is used instead. Setting COTW_RETRY_ATTEMPTS=0 (disable retries) or COTW_RETRY_BACKOFF_MS=0 (no backoff) would be silently ignored.

Cross-file Pattern 1 — eight files now. Most concentrated instance in the entire audit (5 in one block).

Fix: Replace || with ?? on all five lines.

---

### M2 — Realm Hex ID Generation Is Hardcoded and Limited (Lines 547-554)

Access levels 1-10 produce #F00000 through #F00009. These are hardcoded hex IDs, not generated by the hex ID generator. The project core rule is "You NEVER hand-write IDs." These realm IDs bypass hexIdGenerator.js entirely. They may or may not exist in the database.

Fix: Realm IDs should either be registered in hexIdGenerator.js or stored in a database lookup table.

---

### M3 — Entity Search Failure Fall-Through (Lines 882-886, 933-936, 973-976)

When entity search fails in Stages 3, 4, and 5, the catch block logs the error and increments a counter, but then falls through to the next stage silently. A database timeout on a high-confidence strict regex match (Stage 4) falls through to loose keyword matching (Stage 5) and eventually fallback search (Stage 6) — potentially returning a lower-confidence result for what was originally a high-confidence match.

The user gets a wrong confidence rating and potentially a different entity resolution path. This is a functional correctness issue.

Fix: On transient DB failure, either retry at the same stage or return an explicit error state rather than silently degrading to a lower-confidence stage.

## MINOR Findings (5)

### m1 — Regex Compiled Inside Hot Path Methods (Lines 658, 769, 794-795, 833)

Several regex patterns compiled inside methods that run on every matchIntent call. Should be pre-compiled module-level constants. Not a correctness issue but unnecessary GC overhead on a hot path.

---

### m2 — ENTITY_SEARCH_TIMEOUT_MS Defined But Never Used (Line 152)

CONFIG.ENTITY_SEARCH_TIMEOUT_MS is set to 5000ms but is never referenced anywhere in the code. The withRetry wrapper handles retries, but there is no Promise.race timeout on any entity search call. Entity searches can hang indefinitely.

---

### m3 — Duplicate Entity Search Pattern (Lines 868-878, 918-929, 958-969, 993-1003)

The same withRetry + searchEntityWithDisambiguation + result shaping pattern is repeated four times verbatim across Stages 3, 4, 5, and 6. Could be extracted to a helper method.

---

### m4 — FAREWELL speechAct Mapping Wrong (Line 383)

FAREWELL maps to speechAct: 'social.greet'. A farewell is not a greeting. Should likely be 'social.farewell' or a dedicated farewell speech act.

---

### m5 — Logger API Inconsistency (Lines 596, 711, 883, 934, 974, 1009)

Passing raw Error objects as second argument instead of { error: error.message }. If the logger does not handle Error serialisation, this could produce empty or malformed log entries.
