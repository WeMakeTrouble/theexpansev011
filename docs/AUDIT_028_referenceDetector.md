## MAJOR Findings (1)

### M1 — Gazetteer Normalization Mismatch (Line 630 vs taughtEntityCapturer Line 149)

The gazetteer filter on line 630 normalizes candidate phrases with c.phrase.toLowerCase() only — punctuation is preserved. But cotw_user_taught_entities.entity_name_normalized is generated by taughtEntityCapturer._normaliseName() which strips ALL non-alphanumeric characters.

Example: User teaches "O'Connor". taughtEntityCapturer stores entity_name_normalized = "oconnor". Next conversation, referenceDetector detects "O'Connor" again. Gazetteer sends "o'connor" to compare against "oconnor". No match. User gets asked about "O'Connor" again despite already teaching it.

Same issue affects hyphenated names: Jean-Luc becomes "jeanluc" in DB but "jean-luc" in gazetteer check.

Simple names like "Max", "Sarah", "Bondi" are unaffected because toLowerCase() and full normalization produce identical results for plain ASCII names.

Fix: Apply the same normalization function used by taughtEntityCapturer before sending to the gazetteer query. Either import and reuse the normalization, or extract it to a shared utility.

## MINOR Findings (4)

### m1 — Span Format Inconsistency Across Signals

Signal 1 (line 440) produces span: [startIndex, raw.length] where startIndex is the token position in the input. Signals 2-5 (lines 471, 501, 531, 563) produce span: [match.index, match[0].length] which spans the entire regex match including the possessive/preposition prefix. "my Max" in Signal 2 has a span covering "my Max", while Signal 1 for the same "Max" would span only "Max". Downstream consumers processing spans get different coverage depending on which signal detected the entity.

### m2 — Gazetteer Fails Open on DB Error (Lines 676-682)

On any database error, the gazetteer returns all candidates unfiltered. This is documented and intentional (line 624), but means a database failure causes already-known entities to be re-detected and re-asked about. Combined with M1, this creates two paths where the gazetteer does not filter properly.

### m3 — No Validation on userId Format (Line 196)

detectReferences accepts any truthy userId on line 247 without hex format validation. A malformed userId would be passed directly to the three gazetteer SQL queries. The parameterized queries prevent injection, but the queries would return no results, making the gazetteer silently ineffective.

### m4 — Hardcoded Threshold (Line 155)

REFERENCE_THRESHOLD = 0.50 is hardcoded. Cannot tune detection sensitivity via environment variables without code change.

Fix: Move to ENV with nullish coalescing: Number(process.env.REFERENCE_THRESHOLD) ?? 0.50
